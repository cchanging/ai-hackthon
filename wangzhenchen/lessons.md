# 主要收获

第一，项目目标必须拆得足够细。
只有把总体目标持续拆分成阶段目标、组件目标和更小的子目标，AI 才能稳定执行。每一个子目标在进入下一阶段之前，都必须配套生成足够充分的测试用例，把功能、边界条件和异常情况测试清楚；只有测试结果达到预期标准，才能进入下一步。否则，问题会在后续阶段不断累积，最终导致整体质量失控。

第二，代码开发流程需要明确分工为 Plan、Implement、Review 三个阶段。
这三个阶段不应混在一起，而应尽量交给不同的 AI Agent 分别完成：一个负责制定计划和拆解任务，一个负责具体实现代码，一个负责审查实现结果、发现问题并提出修改意见。同时，每一轮都要留下足够详细的文字记录，包括设计思路、实现说明、测试结果和 review 意见。这样做的价值在于，可以通过文档把上下文沉淀下来，减轻 AI Agent 上下文窗口有限带来的影响，使后续迭代能够在已有设计和历史记录的基础上继续推进。

第三，AI 更适合调研和辅助设计，不适合在缺少详细设计的情况下直接生成代码。
在需求和设计不清晰时，AI 直接写代码往往容易偏离目标，生成的实现也不够稳定，后续修改成本很高。相比之下，AI 在资料调研、方案比较、问题拆解、测试设计和代码 review 这些环节更有优势。只有当设计已经足够明确、接口和目标已经定义清楚时，再让 AI 参与代码生成，效果才会更好。
