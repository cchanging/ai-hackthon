# 经验总结

## 对其他开发者最有价值的经验

### 1. 最适合交给 Agent 的，不是“整个项目”，而是“边界清楚的一轮任务”

这次开发里最有效的做法，不是直接说“帮我做一个 lockdep”，而是把工作拆成：

- `plan-and-impl`：按 phase 连续实现主线能力
- `review-round1`：专门提升 `lock class` 精度并修误报
- `review-round2`：专门分析 IRQ 现状、补设计、接 `irq_entries`
- `review-round3`：专门做全仓库扫描、配置裁剪和代码重构
- `review-round4`：专门加 `atomic mode` 规则并做逐案分析

对 Agent 来说，“一轮只做一件大事”远比“一次做完整个系统”更容易产出可靠结果。

### 2. 先给设计文档或现状文档，再让 Agent 开始改代码，质量会高很多

这次开发中，`plan.md`、`lock_class.md`、`interrupts.md` 都起到了关键作用。有效的提问方式不是：

- “帮我想一个方案”

而是：

- “先对照这份文档和现有实现，确认已经做了什么、还缺什么，再继续实现”

这样做的好处是：

- Agent 不容易偏航
- 人和 Agent 对目标边界有共同基准
- 产出更容易和已有设计保持一致

### 3. 对静态分析工具，必须要求 Agent 做真实仓库验证，而不只是跑测试

fixture 测试只能证明“已知场景不回归”，不能证明“真实仓库上的误报和漏报可接受”。

这次最有价值的发现，很多都来自真实扫描，而不是测试：

- `overlayfs` 的显式 `drop(guard)` 误报
- `lock class` 锁类合并过粗的问题
- `wrappers` 为什么会存在，以及能否被分析器能力替代
- `atomic mode` 在依赖 `ostd` 的 crate 中到底命中了哪些真实问题

如果只让 Agent 跑测试，这些问题都不会暴露。

以及我们要找更多的harness,比如说我们在仓库的comment里面定义了一些lock order，这些是可以作为ground truth的，所以我就会要求agent去验证在构建出的lock graph中是否能找到对应的lock sequence。当然最后发现现在并没有办法找到这些lock sequence，这也进一步驱动agent去寻找根因，结果发现是锁的函数间传播存在很大的缺陷，这已经作为了一个limitation。

### 4. 人类必须保留最终判断，Agent并不是完全靠谱

这次实验最清楚的一点是，以下不能完全交给 Agent：

- 误报和真问题的最终判定
- 什么算当前版本必须支持，什么只能诚实写成 limitation
- 是否允许为了通过分析而修改业务代码

尤其在 `review-round3`，用户明确要求“不要改动除了 lockdep 之外的代码”，这个约束非常重要。没有这个约束，Agent 很容易通过改业务代码“绕过去”，但那并不是真正提升分析器。
