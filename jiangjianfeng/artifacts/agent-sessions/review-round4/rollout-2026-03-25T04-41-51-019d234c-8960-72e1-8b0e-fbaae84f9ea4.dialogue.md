### Assistant · final_answer · 2026-03-25 05:08:40 UTC

原因在于两边 `bind` 路径的外层同步原语根本不一样。

`unix datagram` 的 case 3 是这条链：

- [datagram/socket.rs:164](/root/asterinas-codex/kernel/src/net/socket/unix/datagram/socket.rs#L164) 调到 `self.local_receiver.bind(addr)`
- [datagram/message.rs:153](/root/asterinas-codex/kernel/src/net/socket/unix/datagram/message.rs#L153) 先拿 `self.addr.lock()`
  - 这里的 `addr` 是 `SpinLock<Option<UnixSocketAddrBound>>`，定义在 [datagram/message.rs:126](/root/asterinas-codex/kernel/src/net/socket/unix/datagram/message.rs#L126)
- 然后在锁还活着时执行 [datagram/message.rs:159](/root/asterinas-codex/kernel/src/net/socket/unix/datagram/message.rs#L159) 的 `addr_to_bind.bind()?`
- `UnixSocketAddr::bind()` 对路径地址会走到 [unix/addr.rs:24](/root/asterinas-codex/kernel/src/net/socket/unix/addr.rs#L24) 的 `ns::create_socket_file(...)`
- `create_socket_file()` 内部会拿路径解析器的 `RwMutex` 读锁，见 [ns/path.rs:42](/root/asterinas-codex/kernel/src/net/socket/unix/ns/path.rs#L42)

所以 datagram 是明确的：

`SpinLock` 持有中 -> 进入 VFS -> 获取 `RwMutex`

这正是 atomic mode 违例。

`unix stream` 没有类似问题，原因有两层。

第一层，stream 的外层状态锁不是 `SpinLock`，而是 `RwMutex`：

- `UnixStreamSocket.state` 定义在 [stream/socket.rs:35](/root/asterinas-codex/kernel/src/net/socket/unix/stream/socket.rs#L35)
- `bind()` 在 [stream/socket.rs:314](/root/asterinas-codex/kernel/src/net/socket/unix/stream/socket.rs#L314) 里走 `self.state.write().as_mut()`

`RwMutex` 本身就是 sleeping lock，不会把当前上下文带进 atomic mode。  
所以即使它里面再去拿路径解析器的 `RwMutex`，那也不是 “atomic mode 下去睡眠”。

第二层，stream 的地址字段也不是单独的 `SpinLock`：

- `Init.addr` 只是普通 `Option<UnixSocketAddrBound>`，见 [stream/init.rs:24](/root/asterinas-codex/kernel/src/net/socket/unix/stream/init.rs#L24)
- `Init::bind()` 在 [stream/init.rs:37](/root/asterinas-codex/kernel/src/net/socket/unix/stream/init.rs#L37) 里直接改这个字段
- `Connected.addr` 也是普通 `Option<UnixSocketAddrBound>`，见 [stream/connected.rs:24](/root/asterinas-codex/kernel/src/net/socket/unix/stream/connected.rs#L24)
- `Connected::bind()` 在 [stream/connected.rs:101](/root/asterinas-codex/kernel/src/net/socket/unix/stream/connected.rs#L101) 里也是先 `addr_to_bind.bind()?`，之后才 `call_once(...)`

也就是说，stream 没有 datagram 那种“为了保护本地地址，先拿一把 `SpinLock`，再去做命名空间/VFS 操作”的结构。

更具体地说：

- datagram：先持有 `SpinLock`，再 `addr_to_bind.bind()?`
- stream：先持有 `RwMutex`，再 `addr_to_bind.bind()?`

这两条链在 lockdep 的 atomic-mode 规则下结果当然不同。  
前者会报，后者不会报。

补一句：这不代表 stream 的实现一定没有别的锁顺序风险，只是它不属于 case 3 这种 “破坏 atomic mode” 的问题。  
如果你愿意，我可以下一步继续专门检查 UNIX stream socket 的 `bind/connect/listen/accept` 是否存在别的 lockdep 风险。
