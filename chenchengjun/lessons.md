# Lessons Learned

## 关于大模型能力边界

1. 大模型在“审查、验证、比对、归纳”类任务上，往往比在“从零生成完整方案”上更稳定。


2. 大模型感觉还是比较适合一次性执行较为单一的任务。复杂验证任务里，拆分成多个子任务分别交给不同的 agent 来做，再由主 Agent 或人类做最终确认结果，整体效果通常更可靠。

## 关于 workflow 设计

3. 定义 skill 时，应尽量把固定 workflow 写进脚本，而不是留给模型临场决定。
   输入标准化、任务分类、报告初始化、测试选择、结果渲染等步骤越脚本化，模型的思考负担越小，流程随机性也越低。

4. 子 Agent 的任务一定要边界清晰，不能只给“去看看这段改动有没有问题”这种模糊指令。
   最有效的委派方式，是给定明确输入、输出格式、关注点和责任范围，例如只负责一个 review unit、一类 contract，或者一种测试产出。

## 关于 context 管理

5. 复杂任务一定要严格控制 context 用量，否则模型性能会明显下降。
   可以通过任务拆分、子 Agent 并行、把中间结果落盘等方式，把上下文压力从对话窗口转移到结构化 artifact 上。

6. 一旦 context 到达阈值，且输出开始持续偏离预期，应尽快清理上下文或重新起一个新线程。
    如果继续在污染过的上下文里修修补补，模型很容易进入“越改越烂”的状态。

## 关于约束与输出质量

7. 想让模型稳定地产出可审阅结果，必须给它明确的输出 schema。
    `Findings` 的结构、严重级别、证据 ID、文件锚点、置信度这些字段一旦固定，结果会更一致，也更容易被后续流程消费。

8. “证据链”是约束 hallucination 的有效手段。
    要求模型把结论和代码位置、行为依据、对比对象绑定起来，能显著减少空泛判断和拍脑袋式结论。

9. 模型默认倾向于把事情做满，因此需要明确告诉它哪些事不要做。
    如果不限制，它很可能顺手扩展范围、补无关解释、提前跑测试、顺便改代码，从而造成额外 token 消耗和流程偏移。
